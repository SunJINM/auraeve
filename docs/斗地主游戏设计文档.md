# 斗地主 · LLM 真当玩家 —— 设计文档

> 状态：设计稿，待评审
> 范围基线：main 分支聊天型 WebUI（仅聊天 + 登录）；本功能在 `game` 分支开发。
> 一句话目标：让 auraeve 能在 Web 端"开一局斗地主"，由 **LLM 真正作为玩家**参与，规则由后端引擎兜底，体验流畅、AI 不耍赖。

---

## 一、背景与目标

auraeve 目前只能通过对话交流。本功能给它开一个**全新能力面**：智能体能发起一局图形化的斗地主，自己（LLM）作为玩家坐到牌桌上思考、出牌、说话。

- **首发**：斗地主（经典 3 人，1 地主 vs 2 农民）
- **平台**：仅 Web（不涉及 QQ/钉钉等其他渠道）
- **AI 角色**：AI 是真正的"玩家"，出牌决策交给 LLM；规则合法性由后端确定性引擎裁决
- **分两步落地**：
  - **第一版（本设计）**：1 真人 + 2 AI 的完整单局，跑通"聊天发起 → 牌桌页 → AI 作为玩家行动 → 状态实时刷新"整条链路
  - **第二步（仅记一笔，不在本设计展开）**：房间、多真人联机、真人接管 AI 座位、WebSocket、访客身份

---

## 二、范围

### 第一版做

- 完整单局流程：发牌 → 叫/抢地主 → 出牌轮转 → 结算
- 座位构成：1 真人（你）+ 2 AI（LLM 玩家）
- 规则引擎：标准经典规则 + 基础牌型 + 炸弹/王炸翻倍 + 春天判定
- AI 玩家：LLM 决策出牌 + 合法性闭环 + 启发式兜底；默认开启"嘴炮"发言
- 前端：单页牌桌（布局见第八节），手牌选牌、出牌/不出/提示、AI 思考中与嘴炮气泡、结算弹窗
- 发起方式：聊天里说"来一局斗地主"→ 智能体调"开局"工具 → 聊天回一张入口卡片 → 进入牌桌

### 第一版不做（留给第二步）

- 房间 / 房间码 / 多真人同桌 / 真人接管某个 AI 座位
- WebSocket 多端同步、访客昵称登录
- 连续对局、积分榜、战绩持久化
- 花样规则（叫分制、明牌加倍、癞子、四带二变体开关等）

---

## 三、整体架构

后端权威 + LLM 决策。三块新东西：

| 模块 | 位置（建议） | 职责 |
|---|---|---|
| 规则引擎 | `auraeve/games/doudizhu/engine.py` | 牌库、发牌、阶段机、牌型识别/校验/比较、记分。**唯一裁判，确定性、无 LLM** |
| 会话编排 | `auraeve/games/session.py` + `manager.py` | 持有一局状态与三个座位，推进轮次；轮到 AI 时驱动 LLM 决策；产出"状态快照/事件"供推送 |
| AI 玩家 | `auraeve/games/doudizhu/ai_player.py` | 把"该 AI 视角的局面"序列化喂给 LLM，解析结构化动作，做合法性闭环与兜底 |

前端不新增页面，牌桌复用现有右侧滑出抽屉（`FileDrawer`），与聊天并排不遮挡；通信用 **HTTP（发起/真人操作）+ SSE（状态推送）**。

```
聊天"来一局斗地主"
  → 智能体调 start_doudizhu 工具
  → GameManager 创建 GameSession（进程内存）
  → 工具返回入口卡片(game_id) → 聊天渲染卡片
点击入口
  → 前端打开牌桌抽屉（openDrawer，与聊天并排），GET 当前状态 + 订阅 /games/{id}/events(SSE)
轮到 AI：编排器 → AIPlayer.decide()(LLM) → 引擎校验/兜底 → 应用 → 广播快照(含嘴炮)
轮到真人：前端选牌 → POST /games/{id}/actions → 引擎校验 → 应用 → 广播快照
某家出完手牌 → 引擎结算 → 广播结果 → 前端弹结算
```

**为什么这样切**：规则与状态都在后端，前端是"瘦客户端"。第二步做房间时，前端本就是订阅式、后端本就是权威，只需把"单局"升级为"多端房间"、把 SSE 换/补 WebSocket 双向，**前端与引擎几乎不返工**。

## 四、规则引擎设计（确定性，唯一裁判）

引擎不依赖 LLM，纯函数式状态机，便于单测。

### 牌与发牌

- 一副 54 张：3~10、J、Q、K、A、2，加大王、小王
- 牌力顺序：`3 < 4 < … < A < 2 < 小王 < 大王`（比大小用"牌力值"而非点数）
- 发牌：洗牌后每人 17 张，留 3 张底牌；地主确定后底牌并入地主手牌（→ 地主 20 张）

### 阶段机

`待发牌 → 叫地主 → （抢地主）→ 出牌 → 结算`

- **叫/抢地主**：随机定首叫者，轮流选"叫地主/不叫"；有人叫后其余人可"抢/不抢"；末位叫抢者得地主，无人叫则本局流局（重发，可配置）
- **出牌**：地主先出。轮到某家可"出牌"或"不出（过）"；当其余两家连续"过"，最后出牌者获得自由出牌权（可出任意合法牌型）
- **结算**：任一家手牌出完即终局；判定地主胜/农民胜，结合倍数计分

### 牌型（第一版支持）

单张、对子、三张、三带一、三带二、单顺（≥5 连）、双顺（连对 ≥3 对）、飞机（连续三张，带或不带翼）、炸弹（四张同点）、王炸（大小王）、四带二（带两单或两对）。

引擎职责：① 识别一组牌的牌型；② 判断"能否压上一手"（同型更大，或炸弹/王炸压非炸）；③ 合法性校验（牌是否在手中、牌型是否成立、是否压得过）。

### 计分

底分 × 倍数。倍数来源：炸弹/王炸每个翻倍、春天/反春天翻倍。第一版倍数规则固定，不做开关。

## 五、AI 玩家设计（方案三核心）

让 LLM 像人一样打牌，但**绝不允许它违规或拖垮体验**。

### 决策输入（"该 AI 视角"的局面）

序列化为提示词的信息仅限该 AI 应当知道的：自己的手牌、谁是地主、当前要应对的上家牌（或自己可自由出）、本轮已"过"的情况、各家**剩余张数**、台面已出过的牌（公共信息）、当前倍数。**不泄露其他玩家手牌**。

### 决策输出（结构化）

要求 LLM 输出 JSON：动作类型（`play` / `pass`）、要出的牌列表、一句可选"嘴炮"。叫地主阶段输出（`call` / `pass` / `rob` / `no_rob`）。

### 合法性闭环（关键）

```
LLM 产出动作
  → 引擎校验
      合法 → 应用
      非法 → 把"非法原因"回灌，让 LLM 重试（≤2 次）
              仍非法 / 超时 / 解析失败 → 启发式兜底
```

- **启发式兜底**：保证流畅与不卡死。自由出牌时出"最小的单张/最小合法牌型"；需应对时出"刚好压得过的最小牌型"，压不过则"过"。兜底只在 LLM 失败时触发。
- **决策实现**：用现有 provider（`OpenAICompatibleProvider`）做**单次结构化调用**，不套子智能体/工具循环（出牌是单步决策，重型编排只会更慢）。AI 玩家可单独配置模型（默认可用更快的模型以缩短"思考中"等待）。

### 嘴炮（默认开）

LLM 在出牌时附一句性格化短发言，随状态广播到前端气泡。可通过开局参数关闭。嘴炮只增气氛，不参与规则。

## 六、会话编排与数据流

- `GameManager`：进程内 `game_id → GameSession` 映射；负责创建、查询、（第一版）超时回收。
- `GameSession`：持有引擎状态 + 三个 `Seat`（`human` / `ai`）。核心是 `advance()` 推进循环：
  - 轮到 `human`：挂起，等前端 POST 的动作
  - 轮到 `ai`：调 `AIPlayer.decide()` → 引擎校验/兜底 → 应用 → 产出事件
  - 每次状态变化产出一个**状态快照事件**，经 SSE 广播
- **状态快照（推给前端）**：阶段、当前轮到谁、各家剩牌数、台面上一手牌及其出牌者、倍数、地主是谁、（仅本人）自己的手牌、最近一次 AI 嘴炮、结算信息。**绝不把 AI/其他人手牌下发前端。**

### HTTP / SSE 接口（建议，沿用 `/api/webui/` 前缀与 auth 依赖）

| 方法 | 路径 | 用途 |
|---|---|---|
| POST | `/api/webui/games` | 创建一局（也可仅由"开局"工具内部创建）|
| GET | `/api/webui/games/{id}` | 拉取当前状态快照（刷新/断线恢复）|
| GET | `/api/webui/games/{id}/events` | SSE 订阅状态推送（沿用现有 `StreamingResponse` SSE 写法）|
| POST | `/api/webui/games/{id}/actions` | 真人动作：叫/抢地主、出牌、不出 |

真人动作同样过引擎校验，非法则返回错误文案，前端提示，不改变状态。

## 七、后端接入点

- **新增"开局"工具**：继承 `auraeve/agent/tools/base.py` 的 `Tool`，如 `start_doudizhu`。`execute()` 内调 `GameManager` 建局，返回 `ToolExecutionResult`，`content` 为可读说明，`data` 带 `game_id` 供前端渲染入口卡片。经 `tools/assembler.py` 注册进工具集。
- **WebUI 路由**：在 `auraeve/webui/server.py` 注册上表四个路由，复用现有 auth 依赖与 SSE（`StreamingResponse`）模式。
- **状态归属**：第一版游戏状态在 webui 后端进程内存，单用户单局足够；与现有聊天链路解耦，互不影响。

## 八、前端牌桌设计

牌桌**不新增页面、不替换聊天**，而是装进现有的右侧滑出抽屉（`FileDrawer`，见第三节），与聊天**并排不遮挡**——边聊边玩。沿用抽屉现成的开关、右侧滑入动画、宽度拖拽、Esc 关闭、"打开即让位"机制。

### 承载方式

- 抽屉新增一种载荷 `mode: 'game'`（payload 带 `gameId`）；聊天入口卡片点击即 `openDrawer({ mode: 'game', gameId })`，不切页。
- **与文件 / diff 抽屉互斥**：二者共用同一右侧槽位与 store，同一时刻只开一个。游戏中点开某个 diff 会临时盖掉牌桌；关掉再点入口卡片即可回到牌桌——**牌局状态在后端，关抽屉不丢局**（见第十一节），重新 GET 快照即恢复。

### 布局（适配右侧竖条抽屉，约 40% 视口宽，可拖拽调整）

牌桌按窄竖布局自上而下排：

- **顶部**：两个 AI 座位紧凑排——头像、昵称、剩牌数、地主标识；轮到它时显示"思考中…"，出牌后弹**嘴炮气泡**
- **中部**：上家出的牌 + 当前倍数 / 阶段
- **底部（你）**：手牌整排可点选，选中上移高亮；主操作"出牌"，另有"不出 / 提示"；叫地主阶段切换为"叫/抢 地主 / 不叫"按钮
- **手牌横向收纳**：窄宽下 17~20 张排不开，用横向滚动或重叠压缩牌宽；不够时用户可拖宽抽屉缓解
- **结算**：终局在抽屉内弹层显示地主/农民胜负与倍数，提供"关闭"返回聊天（第二步再加"再来一局"）

### 前端结构（贴现状）

- 复用 `useFileDrawer`（`store/fileDrawer.ts`）管开关与 `payload`；在 `FileDrawer.tsx` 的 `mode` 分发里新增 `game` 分支渲染牌桌，聊天入口卡片点击即调 `openDrawer`。
- 新增组件：`GameTable`（牌桌容器 + SSE 订阅）、`PlayingCard`/`HandRow`（牌与手牌）、`SeatPanel`（AI 座位 + 气泡）、`ActionBar`（操作区）、`ResultDialog`（结算）。
- `api/client.ts` 增加建局/拉状态/发动作/订阅事件的方法。

> 第二步（多真人房间）若需要大牌桌 / 全屏体验，可让牌桌脱离抽屉独立成页；第一版抽屉方案不影响该演进。

## 九、从聊天发起

1. 用户："来一局斗地主"
2. 智能体识别意图 → 调 `start_doudizhu` 工具 → 建局得 `game_id`
3. 工具结果在聊天里渲染一张**"进入牌桌"入口卡片**（复用现有 transcript 卡片渲染机制）
4. 点击 → 前端打开牌桌抽屉（`openDrawer`，与聊天并排），开始 GET 状态 + 订阅 SSE

## 十、核心数据结构（示意，非最终签名）

- `Card`：牌力值 + 花色（大小王特判）
- `Move`：`type`(play/pass) + `cards` + 识别出的 `combo`（牌型 + 比较键）
- `Seat`：`kind`(human/ai) + `name` + `hand`(后端持有) + `is_landlord`
- `GameState`：`phase`、`turn`、`landlord`、`multiplier`、`last_move`、各家 `remaining`、`winner`
- `StateSnapshot`（下发前端）：从 `GameState` 投影，剔除他人手牌，附本人手牌与最近嘴炮

---

## 十一、错误处理与边界

- **AI 失败/超时/非法**：重试 ≤2 → 启发式兜底，绝不卡死
- **真人非法操作**：引擎拒绝 + 错误文案，状态不变
- **断线/刷新**：状态在后端，重新 GET 快照即可恢复
- **流局**：无人叫地主 → 重发（可配置；第一版默认重发）
- **单局回收**：终局后保留短时供查看；`GameManager` 定时清理过期内存局

---

## 十二、测试策略

- **规则引擎（重点）**：放 `tests/` 下，单文件独立。覆盖牌型识别、压牌比较、各牌型合法性、发牌张数、叫抢流程、结算与倍数。
- **AI 合法性闭环**：mock LLM 返回（合法 / 非法重试成功 / 连续非法触发兜底 / 超时兜底），断言最终落子始终合法。
- **会话编排**：状态机阶段流转、真人挂起与 AI 自驱推进、快照投影不泄露他人手牌。
- **前端（vitest）**：牌桌渲染、选牌与出牌交互、SSE 快照更新、结算弹层。

## 十三、待确认 / 可调（评审聚焦）

1. AI 玩家用哪个/哪些模型？是否给两个 AI 不同"性格 + 模型"以区分难度？
2. 嘴炮默认开，是否需要在开局参数里提供"关闭嘴炮"开关？
3. 流局是否默认重发（vs 直接结束本局）？
4. 结算后第一版是否需要"再来一局"（否则只"返回聊天"，再来一局走聊天重新发起）？
5. "提示"按钮：用引擎启发式给提示，还是也可借 LLM？（建议第一版用引擎启发式，快且免费）
