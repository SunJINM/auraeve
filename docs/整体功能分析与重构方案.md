# AuraEve 整体功能分析与重构方案

> 分析基线：main 分支 `0acb15d`（调整长时智能体预算与上下文压缩）。
> 测试基线：307 通过 / 8 失败（失败均为依赖系统 `grep` 的"守卫测试"，Windows 环境问题，非逻辑缺陷）。
> 本文档只做分析与方案设计，不涉及代码修改。

---

## 一、项目整体功能地图

### 1.1 核心功能清单

| 功能域 | 模块 | 成熟度 | 说明 |
|---|---|---|---|
| 运行时内核 | `agent_runtime/`（kernel、session_attempt、run_orchestrator、command_queue/scheduler） | 成熟 | 统一命令队列 + 工具循环 + 错误恢复，主/子智能体共用执行内核 |
| 工具引擎 | `agent/tools/`（Read/Write/Edit/Grep/Glob/Bash/web_search/web_fetch/agent/cron/任务工具） | 成熟 | 对标 Claude Code 工具语义，含读后写校验、unchanged stub 等细节 |
| 子智能体 | `subagents/`（executor、react_loop、lifecycle、fork、worktree、SQLite store） | 成熟偏复杂 | sync/async/fork 三种模式、六种 agent 类型、steer 注入、预算控制 |
| 上下文工程 | `agent/context.py`、`agent/engines/`、`agent_runtime/prompt/` | 成熟但分层过度 | 模块化系统提示词 + 两层压缩（工具结果清理 → LLM 摘要） |
| 多渠道接入 | `channels/`（terminal/dingtalk/napcat/webui）+ `runtime_channels.py` | 成熟 | 渠道只做收发翻译，统一入命令队列，支持热启停 |
| 配置系统 | `config/`（schema/doctor/legacy/includes/io/paths） | 成熟偏重 | 校验、迁移、doctor 修复、env 替换、include，schema.py 810 行 |
| 技能系统 | `skill_system/` + `agent/skills.py` | 成熟 | 发现/安装（hub/上传/git）/启停/doctor，提示词注入有上限控制 |
| 记忆系统 | `memory_lifecycle.py` | 成熟（刚简化） | MEMORY.md + 每日日志，靠 Grep/Read 检索，无向量库 |
| 定时与心跳 | `cron/`、`heartbeat/` | 成熟 | cron 经主命令队列路由；心跳空文件跳过模型调用 |
| 语音识别 | `stt/`（bytedance/funasr/openai/whisper-cli 四 provider） | 成熟 | 规范化、缓存、failover |
| MCP | `mcp/` + WebUI mcp_service | 成熟 | stdio/http、健康检查、热重载、审计 |
| WebUI | `webui/`（后端）+ `webui/`（前端 React） | **割裂** | 后端保留 config/mcp/skill/profile 全套 API，前端只剩聊天+登录页 |
| 可观测性 | `observability/`（SQLite 索引 + 订阅推送） | 成熟 | 工具调用/文本流事件均落库并支持 SSE 订阅 |
| 个人资料迁移 | `profile_transfer.py` | 雏形够用 | 配置/记忆/技能/状态打包导入导出 |
| 多模型注册 | `llm/model_registry.py` + `providers/` | 部分死代码 | 运行时只用 OpenAICompatibleProvider；litellm 路线未接入 |

### 1.2 主链路（完整）

```
渠道(napcat/dingtalk/terminal/webui/cron/heartbeat)
  → RuntimeCommandQueue（now/next/later 优先级）
  → RuntimeScheduler（全局串行，单并发）
  → RuntimeKernel.execute_command → _process_message
      → PromptAssembler → LegacyContextEngine → ContextBuilder（系统提示词+记忆+技能）
      → RunOrchestrator（限流/过载/上下文溢出恢复）
      → SessionAttemptRunner（工具循环 + 循环守卫 + 主动压缩 + checkpoint 插队消息）
      → 工具执行（策略引擎判定 → 并发执行 → 结果压缩）
  → SessionManager 持久化 + MemoryLifecycle 日志
  → OutboundDispatcher → 渠道发送
```

子智能体链路：`agent` 工具 → SubagentExecutor → ReActLoop → 复用 SessionAttemptRunner/RunOrchestrator → 完成后以 `task-notification` 命令回注主队列唤醒主智能体。链路完整且有测试覆盖。

### 1.3 成熟与雏形判断

- **已成熟**：工具循环内核、子智能体三模式、渠道接入、压缩策略、技能安装链路、配置 doctor。
- **只是雏形**：profile 迁移（无版本兼容策略）、observability 查询消费端（数据落库但 WebUI 无展示页）、coordinator 类型子智能体（可再派发 agent，但缺深度编排测试）。
- **割裂/维护成本高**：WebUI 前后端不对称；双任务系统（todo vs TaskCreate 四件套）按渠道切换；提示词文本散落多处；配置热更新三条路径。

## 二、架构问题识别

### 2.1 职责过重的文件

| 文件 | 行数 | 问题 |
|---|---|---|
| `agent_runtime/session_attempt.py` | 629 | `run()` 单方法承担：预算、steer、checkpoint、压缩、循环守卫、策略判定、并发执行、观测埋点、结果分类 |
| `agent_runtime/kernel.py` | 613 | 同时负责：组件装配、命令路由、会话命令(/new、/help)、提示词组装调用、输出净化、transcript 持久化、热更新 |
| `observability/manager.py` | 614 | 落库、检索、订阅、保留策略一体 |
| `agent/context.py` | 576 | 提示词所有 section + 消息组装 + 多模态 content block 构造 |
| `skill_system/service.py` | 579 | 列表/安装/启停/doctor/同步全在一个文件，且内嵌 Windows 工具路径探测 |
| `config/schema.py` | 810 | 校验 + 归一化 + WebUI schema 生成三件事 |
| `channels/napcat.py` + `agent/tools/napcat.py` | 588+415 | 渠道协议、媒体解析、TTS、表情映射混在一起 |

### 2.2 模块边界不清晰

1. **提示词组装四层嵌套，抽象只有一个实现**：
   `PromptAssembler → ContextEngine(ABC) → LegacyContextEngine → ContextBuilder`。
   `ContextEngine` 抽象基类只有 `LegacyContextEngine` 一个实现；`PromptAssembler` 的实际增量只有预算报告日志和 runtime_instruction 注入。四层可以压成一层。

2. **上下文压缩入口有三处**，策略分散：
   - `session_attempt._maybe_compact_context()`（每轮调用前主动压缩）
   - `run_orchestrator._compact_messages()`（ContextOverflow 异常恢复）
   - kernel `_process_message` 中处理 `assemble_result.compacted_messages`（组装期压缩回写 session）
   三处各自切分 system/history、各自调 `compaction.py`，行为差异只能靠读代码确认。

3. **配置热更新三条链路、私有属性穿透**：
   `ConfigService.apply → RuntimeHotApplyService.apply → kernel.reload_runtime_config`，且热更新直接改写其他对象私有字段：`policy._global_deny`、`assembler._token_budget`、`agent._channel_users`、`engine._builder`、`engine._memory_window`。任何内部重命名都会静默破坏热更新。

4. **主/子智能体两套外围**：执行内核（SessionAttemptRunner）已统一，但 prompt 构建（ContextBuilder full 模式 vs react_loop 手写文本）、session 持久化（kernel `_persist_tool_transcript` vs executor `_run_task` 末尾手动回写）、预算策略（主强制无限 vs 子按 TaskBudget）各搞一套。

5. **WebUI 前后端不对称**：后端维护约 30 个 config/mcp/skills/profile 路由 + 全套 pydantic schema + `build_webui_schema_groups()`，前端已在"精简前端界面"提交中删掉对应页面，仅剩聊天。这是当前最大的"维护成本悬空"。

### 2.3 重复逻辑

| 重复点 | 位置 |
|---|---|
| 预算耗尽汇总两段逐字相同（约 30 行 ×2） | `session_attempt.py` `run()` 内 turn 预算分支与 tool-call 预算分支 |
| 子智能体提示词文本两份（执行约束十几条逐字重复） | `react_loop._build_system_prompt()` 与 `_build_agent_context()+_build_runtime_instruction()`；前者仅在 `prompt_assembler=None` 时走到，而 executor 永远传入 assembler——**死路径** |
| 消息构造 helper 两套 | `session_attempt._add_assistant_msg/_add_tool_result` 与 `context.py add_assistant_message/add_tool_result`（后者疑似无调用方） |
| SSE 端点两个完全相同 | `webui/server.py` `/chat/events` 与 `/chat/transcript/events`，处理函数逐字一致 |
| 记忆实现两套 | `memory_lifecycle.py`（在用）与 `agent/memory.py` MemoryStore（**零引用，死代码**） |
| 热更新核心键处理 | `kernel.reload_runtime_config` 与 `RuntimeHotApplyService` 对 CORE_RUNTIME_KEYS 的拆分边界重叠 |

### 2.4 耦合过高

- kernel 在构造期把 `self._subagent_executor` 传给 `build_tool_registry` 的 lambda 里又引用 `self._subagent_executor`（自引用闭包），且 WebUI server 直接拿 `agent._subagent_executor` 私有属性。
- `runtime_hot_reload._reload_skills()` 通过 `hasattr(engine, "_builder")` 反射链取 SkillsLoader。
- `webui_server._restart_callback` 在 main.py 中事后赋值私有属性。

### 2.5 可删除的旧功能/兼容逻辑

| 项 | 证据 |
|---|---|
| `providers/litellm_provider.py` + `providers/registry.py`（约 300 行） | 运行时唯一构造点是 `main._build_provider()`，只用 OpenAICompatibleProvider；litellm 无任何 import 方 |
| `agent/memory.py`（MemoryStore） | 全仓库零引用 |

| 仓库内 `.subagents/*.jsonl`、`__pycache__/`（含已删测试的 .pyc） | 运行时产物入库，应清理并补 .gitignore |

## 三、代码精简机会汇总

1. **可直接删除（死代码）**：见 2.5 表格，合计约 1500+ 行 Python 与一个死依赖。
2. **可合并**：
   - 四层 prompt 组装 → 单一 ContextPipeline；
   - 两个 SSE 端点 → 一个；
   - 两套消息构造 helper → 一处；
   - kernel.reload_runtime_config 并入统一热更新协议。
3. **可抽象的重复**：预算耗尽汇总、子智能体执行约束文本（单一常量源）、三处压缩入口（统一 Compactor 服务）。
4. **可拆分的大文件**：session_attempt（循环骨架 / 工具批执行 / 守卫与预算）、kernel（装配 / 消息处理 / 持久化）、schema.py（校验 / 归一化 / WebUI schema 生成）、napcat 渠道（协议 / 媒体 / TTS）。
5. **可简化的数据流**：
   - 双任务系统二选一（`task_mode.py` 的 task_v2 / legacy_todo 分叉、`legacy_todo_state.py`、`agent/plan.py`、`tools/plan.py` 一并收敛）；
   - 配置热更新统一为"子系统注册式"协议；
   - WebUI 前后端能力对齐（恢复管理页或裁剪后端 API，二选一）。

## 四、成熟方案对标

### 4.1 OpenClaw（最直接的对标物）

OpenClaw 是自托管个人 AI 助手网关：单一 Gateway 进程作为"会话、路由、渠道连接的唯一事实源"，连接 WhatsApp/Telegram/Discord 等十余渠道。AuraEve 的 SOUL.md/AGENTS.md/HEARTBEAT.md、技能目录、心跳协议明显源自它（代码注释中也多处写明"对标 openclaw"）。

- **它解决的问题**：多渠道统一接入、单进程状态权威、设备配对信任模型、技能运行时发现。
- **适合借鉴**：
  - **技能选择性注入**——OpenClaw 不把所有技能塞进每轮 prompt，只注入与当前轮相关的技能。AuraEve 当前是全量注入 + 字符上限截断，技能多时会浪费预算，可引入按需匹配。
  - **runId + 幂等键的渠道协议**——AuraEve WebUI 已有 idempotencyKey，可推广到其他渠道的副作用操作。
- **不适合**：设备/节点配对、多客户端 WS 控制面。AuraEve 已删除远程节点系统，这是正确决策，不应回头重建。

### 4.2 OpenAI Agents SDK / Responses API

核心抽象是 Agent（指令+工具）、Runner（内置 agent loop）、Sessions、Handoffs（代理间移交）、Guardrails（输入输出校验）、Tracing。它的价值在于**把 agent loop 收敛为单一可信实现**，开发者只组合不重写。

- **适合借鉴**：
  - "Runner 单一循环"理念佐证了 AuraEve 让主/子共用 SessionAttemptRunner 的方向是对的，下一步应把循环外围（prompt、持久化、预算来源）也统一参数化，而不是两套包装。
  - **Guardrails 位置**：AuraEve 的 ToolPolicyEngine 只管工具调用，缺输出侧校验（当前 `_sanitize_assistant_output` 是硬编码在 kernel 里的特例），可归一为"输出守卫"扩展点。
  - **Tracing 模型**：RunTrace 已存在，但只随结果返回；可像 SDK 一样与 observability 打通成统一 trace 视图。
- **不适合**：Handoffs（对话所有权移交）。AuraEve 是单主智能体+工作型子智能体模型，移交语义会复杂化渠道路由。

### 4.3 AutoGen / Microsoft Agent Framework、CrewAI

MS Agent Framework（AutoGen 后继）提供 sequential/concurrent/handoff/group-chat 编排、持久化工作流、人审环节；CrewAI 以 Crew/Agent/Task/Process 做角色化分工，附带任务结果缓存与断点恢复。

- **它们解决的问题**：企业级多智能体编排的可靠性（durable workflow）、角色分工的声明式表达。
- **适合借鉴**：
  - CrewAI 的**任务结果缓存/断点恢复**思想：AuraEve 子智能体失败后只能重派，SQLite store 里已有 transcript，可支持"从已有上下文 continue"作为轻量恢复（continue 已有，但缺失败后自动续跑策略）。
  - 声明式 agent 定义：AuraEve 的 `agents/definitions.py` 已是此模式，保持即可。
- **不适合**：图式工作流引擎、group-chat 协商。个人助手场景的任务多为"一主多从汇总"，coordinator 类型已覆盖；引入工作流引擎是过度设计。

### 4.4 Claude Code / Continue / Cursor（代码 Agent 工作流）

Claude Code 的关键机制：子智能体独立上下文窗口 + fork 继承模式、技能按需加载（不常驻 prompt）、hooks 架构性拦截（不可被模型"说服"绕过）、自动压缩（先清旧工具输出、再摘要）、任务清单工具。

- **适合借鉴**：
  - AuraEve 的 fork/sync/async、两层压缩、unchanged-stub Read、TaskCreate 四件套都已对标实现且质量不错——这部分应**保留并固化为边界**，不要再加变体。
  - **hooks 的"架构性强制"**：ToolPolicyEngine 已有 evaluate/rewrite，可补一个用户可配置的 pre/post 钩子声明层（配置驱动而非代码注册），替代散落的硬编码拦截。
  - **子智能体持久记忆目录**（Claude Code subagent memory 字段）：对 explore 类智能体跨次积累认知有价值，可列为远期。
- **不适合**：IDE 集成、checkpoint/rewind 文件快照（Cursor/Claude Code 的编辑器场景），聊天助手场景收益低。

### 4.5 对标结论

AuraEve 的架构方向（单进程内核、统一命令队列、共享工具循环、两层压缩、声明式子智能体）与成熟方案一致，**问题不在缺能力，而在"同一能力存在多套实现/多个入口"**。重构主题应是收敛与删减，而非引入新框架。

## 六、分阶段重构方案

总原则：每阶段独立可合入、可回滚；先删后并、先并后拆；任何阶段不改变对外行为（CLI 命令、WebUI API 契约、渠道消息语义），除非该阶段目标就是裁剪契约。

### 阶段一：死代码与仓库卫生清理

- **改造目标**：删除全部零引用代码、死依赖、运行产物，修复文档失链，使后续阶段的 diff 干净可审。
- **涉及模块**：
  - 删除：`providers/litellm_provider.py`、`providers/registry.py`、`agent/memory.py`、`node_subagent/`、`runtime_hot_reload.sync_message_tool_settings`、`kernel.register_channel_sender`、`react_loop._build_system_prompt/_prepare_messages` 死路径、8 个守卫测试。
  - 依赖：requirements.txt / pyproject.toml 移除 playwright，更新过期注释（vector memory、browser tool）。
  - 仓库卫生：清理 `.subagents/*.jsonl`、各处入库的 `__pycache__`，补 .gitignore 规则。
  - 文档：README 的 docs/README.md 失链与"节点系统文档"导航修正；`docs/superpowers/` 历史计划文档归档说明。
- **预期收益**：约 1500+ 行死代码与一个重型死依赖出库；测试在 Windows 上恢复全绿；新人阅读面缩小。
- **风险点**：极低。唯一需确认 litellm 路线是否属于"规划中未接入"——若是规划能力，改为移到独立分支保存而非删除。
- **测试/验证方式**：全量 pytest（307 项应全过，且不再有 grep 依赖失败）；`python -m auraeve run --terminal` 冒烟；`pip install -e .` 后 import 扫描（如 `python -X importtime -c "import auraeve"`）确认无残余引用。

### 阶段二：重复逻辑收敛

- **改造目标**：同一逻辑只保留一份实现，消除"改一处漏一处"的隐患。
- **涉及模块**：
  - `session_attempt.py`：两段预算耗尽汇总抽成单个内部函数（提示词文本作为模块级常量）。
  - `subagents/runtime/react_loop.py`：执行约束文本单源化（与阶段一删除死路径配合后，仅剩 `_build_agent_context`/`_build_runtime_instruction` 两个消费方共享同一常量列表）。
  - `webui/server.py`：合并两个 SSE 端点（保留两个路由路径也可，但处理函数只留一个）。
  - 消息构造 helper：`context.py` 的 `add_tool_result/add_assistant_message` 若确认无调用方则删除，统一使用 `session_attempt` 模块级函数。
  - `agent` 工具：移除 `run_in_background` 兼容参数（提示词与 schema 同步更新）。
- **预期收益**：子智能体提示词调整从"改三处"变"改一处"；预算分支行为保证一致。
- **风险点**：提示词文本合并时若措辞有意差异（system prompt 版与 runtime_instruction 版），需逐行 diff 确认语义等价；`run_in_background` 删除影响已写入会话历史的旧工具调用回放（仅影响模型重试场景，低风险）。
- **测试/验证方式**：现有 `test_subagent_prompts.py`、`test_subagent_budget_summary.py`、`test_agent_tool.py` 全过；新增断言"约束文本常量被两个构建函数共同引用"的单测；WebUI SSE 用 preview/curl 验证事件流不变。

### 阶段三：上下文组装层扁平化与压缩统一

- **改造目标**：四层 prompt 组装压成一层；上下文压缩收敛为单一服务，三个调用点共享同一策略与日志口径。
- **涉及模块**：
  - 合并 `agent_runtime/prompt/assembler.py`、`agent/engines/base.py`、`agent/engines/legacy.py` 进入一个 `ContextPipeline`（落点建议 `agent_runtime/prompt/`），`ContextBuilder` 作为其内部 section 渲染器保留。
  - `ContextEngine` ABC 删除；kernel 与 SubagentExecutor 均改持 ContextPipeline。
  - 新建 `ContextCompactor`（包装现 `engines/compaction.py`），统一"should_compact → clear_tool_results → LLM 摘要"流程与 system/history 切分；`session_attempt._maybe_compact_context`、`run_orchestrator._compact_messages`、kernel 组装期压缩三处改为调用它。
  - 热更新中对 `assembler._token_budget`、`engine._memory_window` 的私有写入改为 ContextPipeline 公开方法（为阶段五铺垫）。
- **预期收益**：prompt 链路调用栈从 4 层降到 1–2 层；压缩行为单点可调可观测；token 预算来源唯一。
- **风险点**：本阶段是行为敏感区——压缩触发阈值、压缩后回写 session 的时机若有偏差会改变长会话行为。需保持"组装期压缩回写 session、轮内压缩仅影响送模型上下文"这两条现有语义不变。
- **测试/验证方式**：`test_prompt_assembler_extra.py`、`test_session_attempt_tool_result_compaction.py`、`test_kernel_*` 全过；新增 ContextCompactor 三场景单测（不压缩 / 仅清工具结果 / 摘要兜底）；构造超预算长会话做端到端回归，对比压缩前后 token 估算日志。

### 阶段四：任务系统统一

- **改造目标**：消除 legacy_todo / task_v2 双轨，全渠道统一为 TaskCreate/TaskGet/TaskUpdate/TaskList（task_v2）。
- **涉及模块**：
  - 删除：`agent/tools/plan.py`（TodoTool）、`agent/plan.py`（PlanManager）、`agent/legacy_todo_state.py`、`agent_runtime/task_mode.py` 的渠道分叉与 `AURAEVE_ENABLE_TASKS` 环境变量。
  - 调整：`kernel._resolve_runtime_tools`（去掉 extract_latest_todos 回放）、`kernel._inject_plan_into_messages`、`tools/assembler.register_task_tools`、子智能体 tool_builder 中 `task_mode="legacy_todo"` 的传参、ContextBuilder 中 todo 相关提示词分支。
  - 子智能体默认任务模式需明确决策：子体短任务可不挂任务工具（保持现状 is_subagent→False 的语义，即直接不注册任务工具）。
- **预期收益**：一套任务心智模型；kernel 每轮少一次 session 全量扫描（extract_latest_todos）；提示词中任务指引唯一。
- **风险点**：napcat/dingtalk 渠道此前走 legacy_todo，切到 task_v2 后任务以文件落盘（state_dir/tasks），需确认非交互渠道的任务列表不会无限累积——应在 TaskStore 现有的过期清理逻辑上验证；旧会话历史中的 todo 工具调用记录回放时模型可能尝试调用已删工具（工具不存在时返回错误文案即可，属可接受降级）。
- **测试/验证方式**：改写 `test_kernel_task_modes.py`、`test_task_mode_and_tools.py` 为单轨断言；`test_legacy_todo_state.py` 删除；钉钉/QQ 渠道手动冒烟一次多步任务，确认任务提醒注入（task_reminders）正常。

### 阶段五：配置热更新协议统一与内核瘦身

- **改造目标**：热更新从"中心服务反射穿透各对象私有字段"改为"子系统注册 reload 接口"；kernel 拆出消息处理与持久化职责。
- **涉及模块**：
  - 热更新：定义统一的 `apply(keys, config) -> {applied, requiresRestart, issues}` 子系统接口；kernel、heartbeat、stt、channels、skills、mcp 各自实现；`RuntimeHotApplyService` 退化为注册表+分发器；`kernel.reload_runtime_config` 并入 kernel 自己的实现；消除对 `policy._global_deny`、`agent._channel_users`、`engine._builder` 等私有写入（各对象提供公开 setter 或 reload 方法）。
  - kernel 拆分：`RuntimeKernel`（装配与生命周期）/ `MessageProcessor`（_process_message、输出净化、会话命令）/ `TranscriptPersister`（transcript 与 memory 回写）。WebUI server 对 `agent._subagent_executor` 的私有访问改为构造期显式传参。
  - 顺带：`config/legacy.py` 标注日落（保留迁移但在 doctor 输出 deprecation 提示，下个大版本删除）。
- **预期收益**：热更新可测试、可扩展（新增子系统只需注册）；kernel 降到 300 行以内；私有属性穿透清零，内部重构不再威胁热更新。
- **风险点**：热更新语义回归面广（CORE/HEARTBEAT/CHANNEL/STT/SKILL/MCP 六组键），必须逐键对照现有行为表迁移；kernel 拆分注意 `_drain_checkpoint_messages` 与 scheduler 的回调闭包不要断。
- **测试/验证方式**：`test_runtime_hot_reload.py`、`test_kernel_orchestrator_wiring.py`、`test_kernel_resume.py` 全过并按新结构补齐；为每个子系统 reload 接口写独立单测；WebUI `config/apply` 端到端验证（改 LLM_TEMPERATURE、HEARTBEAT_ENABLED、NAPCAT_ALLOW_FROM 三类代表键）。

### 阶段六：WebUI 能力对齐与测试补强

- **改造目标**：消除前后端不对称；补齐核心链路缺失的集成测试;固化模块边界。
- **涉及模块**：
  - WebUI 二选一决策（需产品判断，方案默认推荐 A）：
    - **方案 A（推荐）**：前端恢复"配置 / 技能 / MCP"三个管理页（此前已有实现可从 git 历史找回设计），后端 API 不动；
    - **方案 B**：裁剪后端 config/mcp/skills/profile 路由与 schemas（保留 CLI 等价能力），WebUI 定位纯聊天。
  - 测试补强：kernel 消息处理端到端（含静默令牌、meta 事件、cron 投递）集成测试；fork 子智能体继承+回注完整链路测试;压缩长会话回归测试；Windows 兼容性（搜索类工具在无 rg/grep 环境的降级路径）。
  - 边界固化：为 `agent_runtime`、`subagents`、`channels`、`webui` 写明模块契约（允许的依赖方向），可选引入 import-linter 之类的依赖方向检查纳入 CI。
- **预期收益**：WebUI 维护成本与实际能力匹配；核心链路有回归网，后续重构不再依赖人工冒烟。
- **风险点**：方案 A 需要前端工作量且要对齐现有 schema 分组接口；方案 B 是对外契约裁剪，若有外部脚本调用这些 API 会破坏（需先确认无消费方）。
- **测试/验证方式**：前端 vitest（现有 transcript 测试基础上补管理页测试）+ Claude Preview 手动核对三个管理页;后端 pytest 全量;依赖方向检查纳入 CI 后跑一次全仓校验。

## 七、阶段依赖与裁剪建议

```
阶段一(清理) → 阶段二(去重) → 阶段三(上下文扁平化) → 阶段五(热更新协议)
                                  ↘ 阶段四(任务统一,可与三并行)
阶段六(WebUI+测试) 依赖一、二完成,其余可并行推进
```

若需要压缩范围：阶段一、二是纯收益必做；阶段三、五是结构收益最大的两步；阶段四、六可按产品优先级延后。
