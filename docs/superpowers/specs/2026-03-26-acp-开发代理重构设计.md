# AuraEve ACP 开发代理重构设计

## 1. 背景与目标

当前 `auraeve` 已具备以下基础能力：

- 多渠道消息接入
- 本地工具调用
- WebUI 管理界面
- 子体编排与远程节点
- 技能、插件、MCP、日志与配置体系

但如果目标是复刻 `openclaw` 在 `Codex / Claude Code / Gemini CLI` 这类开发代理场景中的效果，当前架构存在几个关键问题：

- `main.py` 承担了过多装配职责，平台入口、运行时、WebUI、节点服务耦合过深
- 会话系统仍以 JSONL 对话历史为中心，无法表达 ACP 开发线程、工作区绑定、工件和运行事件
- 开发能力散落在普通聊天、工具调用、子体编排中，没有独立的开发代理领域模型
- WebUI 当前更偏向普通聊天管理，而不是开发会话观测和接管
- 子体结果默认回注到母体聊天上下文，这更适合研究型助手，不适合稳定的开发线程

本次重构目标不是把 `auraeve` 变成纯开发平台，而是在保留个人 AI 助手平台能力的前提下，新增一个完整、可扩展、可观测的开发代理子系统，并以 ACP 作为第一入口。

## 2. 设计原则

### 2.1 产品原则

- `auraeve` 继续是主平台，不转型成单一用途开发工具
- 开发代理作为一等子系统存在，不再以普通聊天模式伪装
- 优先追求完整功能、扩展性和可用性，不保留无意义兼容层
- 不再使用的旧路径和旧抽象应直接删除，保持工程整洁

### 2.2 架构原则

- 平台层与运行时层分离
- 会话实体与消息历史分离
- 开发会话、普通聊天、子体任务使用统一会话模型，不再各自为政
- 所有运行都通过统一 runtime registry 启动
- ACP 是独立传输入口，不依附于 WebUI
- 开发链路中的所有动作事件化，供 ACP 和 WebUI 共用

## 3. 目标能力

重构完成后，系统应同时具备两类核心能力。

### 3.1 主平台能力

- 保留现有多渠道聊天能力
- 保留插件、技能、MCP、日志、配置、节点管理能力
- 保留定时任务、消息转发、非开发型日常助手能力

### 3.2 开发代理能力

- ACP 客户端可稳定连接 `auraeve`
- 一个开发线程可以长期存在，并与工作区稳定绑定
- 支持文件、命令、浏览器、网页、补丁、工件、审批等开发动作
- 支持开发线程恢复、重连、聚焦、终止、继续
- 开发线程可派生并行子任务，但不会破坏主线程语义
- WebUI 可查看开发会话、事件流、patch、artifact、审批与节点状态

## 4. 目标架构

### 4.1 总体分层

建议将系统重构为五层。

#### 第一层：平台层

职责：

- 配置加载与热更新
- 模型与供应商管理
- MCP、插件、技能管理
- 认证、日志、审计
- 通用基础服务注册

保留并继续复用的模块：

- `auraeve/config`
- `auraeve/providers`
- `auraeve/plugins`
- `auraeve/skill_system`
- `auraeve/mcp`
- `auraeve/observability`

#### 第二层：会话域

职责：

- 管理统一会话实体
- 建立 session key 与 runtime 的映射
- 管理 thread、workspace、agent、run 的绑定关系
- 提供恢复、查询、归档能力

会话不再只是 JSONL 对话记录，而是结构化实体。

#### 第三层：运行时层

职责：

- 统一创建和管理各种运行时
- 按 runtime 类型分配执行模型
- 暴露统一的 run 启动、停止、恢复接口

运行时建议分为：

- `native_runtime`
- `acp_runtime`
- `subagent_runtime`

#### 第四层：开发代理域

职责：

- 开发会话建模
- ACP 线程绑定
- 工作区绑定
- patch、exec、artifact、approval、focus 的统一管理
- transcript、事件流、恢复与审计

这是本次重构的核心新增领域，不再只是一些 coding tools 的堆叠。

#### 第五层：管理与观测层

职责：

- WebUI 管理页面
- 开发会话列表与详情
- ACP 连接状态与事件观测
- 审批、patch、artifact、命令执行记录
- 节点、子任务与日志审计

WebUI 不再是开发主入口，而是开发代理的控制台。

## 5. 统一会话模型

### 5.1 会话类型

建议统一支持以下类型：

- `chat`
- `dev_acp`
- `task`
- `system`

### 5.2 核心字段

每个会话至少包含：

- `session_id`
- `session_key`
- `session_type`
- `runtime_type`
- `agent_id`
- `workspace_id`
- `thread_id`
- `state`
- `created_at`
- `updated_at`
- `metadata`

### 5.3 会话与历史的关系

消息历史不再等于会话本体。会话本体之外，还要有独立事件流。

一个开发会话中需要记录的事件包括：

- 用户输入
- 模型输出
- 工具调用开始与结束
- 文件改动
- patch 提议与应用
- 命令执行开始与结束
- 工件生成
- 审批请求与审批结果
- 运行中断与恢复

JSONL 历史可以继续作为轻量 transcript 存储格式之一，但不能再充当完整会话模型。

## 6. ACP 开发代理设计

### 6.1 ACP 作为第一入口

本次设计明确采用 ACP 作为开发代理第一入口，原因如下：

- 最接近 `openclaw` 对 IDE 和开发客户端的真实接入方式
- 更适合长期开发线程，而不是一次性聊天请求
- 更容易支持 thread 恢复、重连、聚焦和 IDE 上下文绑定
- WebUI 不必承担浏览器内 IDE 的复杂职责

### 6.2 ACP Bridge

建议新增独立 ACP 桥接模块，职责如下：

- 通过 stdio 与 ACP 客户端通信
- 将 ACP 会话映射到内部 `dev_acp` session
- 将客户端 prompt、取消、中断、恢复请求转发到 `auraeve`
- 将内部事件流转译成 ACP 可消费的结构

### 6.3 会话绑定规则

建议采用稳定 session key：

`dev:<agent_id>:<workspace_id>:<thread_id>`

语义：

- 同一个工作区下可以有多个开发线程
- 同一个线程可跨客户端断线重连
- 线程始终绑定到特定工作区与 agent

### 6.4 开发事件流

ACP 与 WebUI 使用同一套开发事件模型。

建议事件类型包括：

- `user_prompt`
- `assistant_output`
- `tool_call_started`
- `tool_call_finished`
- `patch_proposed`
- `patch_applied`
- `command_started`
- `command_finished`
- `artifact_created`
- `approval_requested`
- `approval_resolved`
- `run_aborted`
- `run_completed`

这样可以避免 ACP 链路和 WebUI 链路各自维护一套状态。

## 7. 开发代理领域模型

建议新增以下领域实体。

### 7.1 Workspace

字段建议：

- `workspace_id`
- `path`
- `repo_root`
- `default_branch`
- `active_branch`
- `status`
- `metadata`

职责：

- 绑定开发会话
- 记录仓库信息
- 管理工作区状态与分支信息

### 7.2 DevSession

字段建议：

- `session_id`
- `workspace_id`
- `thread_id`
- `agent_id`
- `runtime_type`
- `focus_state`
- `status`

职责：

- 作为开发线程实体
- 管理长生命周期上下文

### 7.3 Run

字段建议：

- `run_id`
- `session_id`
- `runtime_type`
- `status`
- `started_at`
- `ended_at`
- `trigger`

职责：

- 记录每次执行轮次
- 关联执行事件与资源消耗

### 7.4 Artifact

字段建议：

- `artifact_id`
- `session_id`
- `run_id`
- `kind`
- `path`
- `label`
- `metadata`

职责：

- 记录 diff、报告、日志摘录、图片、下载文件等工件

### 7.5 PatchSet

字段建议：

- `patch_id`
- `session_id`
- `run_id`
- `files`
- `status`
- `created_at`
- `applied_at`

职责：

- 记录补丁提议与应用过程
- 支持审计与回放

### 7.6 ExecutionRecord

字段建议：

- `execution_id`
- `session_id`
- `run_id`
- `command`
- `cwd`
- `exit_code`
- `stdout_summary`
- `stderr_summary`

职责：

- 记录命令执行历史
- 支持复盘、审计和失败分析

### 7.7 ApprovalRequest

字段建议：

- `approval_id`
- `session_id`
- `run_id`
- `action_type`
- `risk_level`
- `status`
- `resolved_by`

职责：

- 统一高风险操作审批
- 为开发会话和子体任务共用

## 8. 运行时设计

### 8.1 runtime registry

建议增加统一运行时注册中心，对外提供统一能力：

- 根据 `runtime_type` 解析运行器
- 启动 run
- 终止 run
- 恢复 run
- 查询运行状态

### 8.2 native runtime

职责：

- 处理现有普通聊天、普通 agent 工具调用
- 面向消息渠道和非开发会话

### 8.3 acp runtime

职责：

- 面向开发会话
- 驱动 coding tool、artifact、patch、approval、workspace 操作
- 维持开发 transcript 与事件流

### 8.4 subagent runtime

职责：

- 负责并行检索、后台任务、慢任务、远程节点任务
- 作为开发会话的旁路执行器

### 8.5 运行时关系

开发链路中：

- 主开发线程由 `acp_runtime` 驱动
- 并行旁路工作由 `subagent_runtime` 驱动
- 子体结果直接回写开发 transcript

不再默认：

- 把子体结果先注入母体聊天
- 再由母体 LLM 改写回复用户

该模式保留给普通聊天和普通任务链路，不再作为开发链路默认行为。

## 9. 工具与策略设计

### 9.1 工具分组

建议将开发工具按组管理，而不是只按名字散列：

- `fs`
- `exec`
- `web`
- `browser`
- `workspace`
- `artifact`
- `approval`
- `coordination`

### 9.2 策略维度

工具策略建议同时考虑：

- `runtime_type`
- `session_type`
- `agent profile`
- `risk_level`
- `workspace policy`

### 9.3 审批策略

以下动作应优先纳入审批：

- 跨工作区写入
- 高风险 shell 命令
- 远程节点上的高风险操作
- 大规模文件修改
- 分支切换、提交、推送类操作

## 10. WebUI 定位调整

WebUI 不再承担主要开发入口，而是承担以下职责：

- 查看开发会话列表
- 查看 session 事件流与 transcript
- 查看 patch、exec、artifact、approval
- 手动中断、恢复、聚焦开发线程
- 查看 ACP 客户端连接与工作区绑定
- 查看子体与节点活动

现有普通聊天 WebUI 保留，但降级为平台附属功能，不再充当开发代理核心入口。

## 11. 对现有代码的处理建议

### 11.1 保留的模块

- `auraeve/config/*`
- `auraeve/providers/*`
- `auraeve/plugins/*`
- `auraeve/skill_system/*`
- `auraeve/mcp/*`
- `auraeve/observability/*`
- `auraeve/subagents/control_plane` 中的调度、审批、telemetry、capability 相关逻辑
- `auraeve/agent/tools/*` 中真正通用的工具实现

### 11.2 必须重写的模块

- `main.py`
- `auraeve/session/manager.py`
- 当前运行时装配方式
- 当前开发链路中的会话与事件表达
- WebUI 中以普通聊天承接开发代理的路径

### 11.3 建议清理的路径

- 把开发代理继续等同为 `webui chat` 的设计
- 开发链路中默认“子体结果注入母体聊天再解释”的模式
- `main.py` 中持续堆叠的总控式装配逻辑
- 不再使用的旧 session 兼容语义

## 12. 建议的新目录结构

建议逐步迁移到如下结构：

```text
auraeve/
  app/
    bootstrap.py
    container.py
    runtime_registry.py
  domain/
    sessions/
    runs/
    workspaces/
    artifacts/
    approvals/
  runtimes/
    native/
    acp/
    subagent/
  transports/
    acp/
    webui/
    channels/
  services/
    session_service.py
    run_service.py
    workspace_service.py
    artifact_service.py
  policies/
    tool_policy/
    runtime_policy/
```

该目录结构的核心目的不是追求形式整齐，而是明确边界：

- 启动装配归 `app`
- 领域模型归 `domain`
- 执行器归 `runtimes`
- 协调能力归 `services`
- 对外入口归 `transports`

## 13. 迁移策略

建议分五个阶段进行。

### 阶段一：搭建新骨架

- 建立新的 `app/container/runtime registry`
- 建立统一 session / run / workspace 模型
- 先不改变现有对外行为

### 阶段二：落地 ACP runtime

- 新增 ACP bridge
- 打通 ACP thread 与内部 session 的映射
- 建立开发事件流
- 支持本地工作区绑定

### 阶段三：接管开发工具链

- 将 coding 相关工具接入 `acp_runtime`
- 建立 patch、exec、artifact 审计
- 插入审批节点

### 阶段四：整合子体能力

- 允许开发会话派生子任务
- 子体结果直接回写开发 transcript
- 保留节点系统作为旁路能力

### 阶段五：切换 WebUI 与清理旧代码

- 新增开发管理页面
- 降级旧的开发聊天入口
- 删除无用兼容层和废弃路径

## 14. 成功标准

本方案完成后，系统应满足以下标准：

- ACP 客户端可以稳定连接并驱动 `auraeve`
- 开发线程具备可恢复、可观测、可审计能力
- 工作区、分支、工件、命令执行、审批都有结构化记录
- 子体可以为开发线程服务，但不会破坏线程语义
- WebUI 可作为开发代理控制台使用
- 旧的无用开发路径被清理，代码结构显著更清晰

## 15. 开发分支建议

本次方案建议在独立开发分支上实施。

推荐分支名：

- `feature/acp-dev-runtime`

该分支用于承载：

- 设计文档
- 实施计划
- 架构重构
- WebUI 调整
- 旧代码清理

## 16. 结论

本次重构的本质不是“给现有 agent 多加几个 coding tool”，而是为 `auraeve` 增加一个独立的开发代理子系统，并将 ACP 提升为开发入口。

通过该方案，`auraeve` 可以继续保留个人 AI 助手平台的优势，同时获得接近 `openclaw` 的开发代理能力，并为后续接入更多开发客户端、运行时和节点协作模式留下清晰扩展空间。
