# 外部编码智能体接入设计方案

## 背景

`auraeve` 当前已经具备较完整的主智能体运行时、工具系统、子体编排器和远程节点机制，但“编码能力”仍主要来自主智能体自身配合本地工具执行。对于以下类型的任务，现有模式存在明显短板：

1. 长时编码任务需要持续的会话状态、工作目录和中断恢复能力。
2. 代码实现、代码审查、方案推演这类任务更适合交给专门的 coding agent。
3. 后续如果继续接入更多外部 coding harness，逐个直连会导致接口和错误语义越来越分裂。

目标不是把 `Claude Code -> Codex -> Claude Code` 之类固定流程写死，而是把 `Claude Code` 和 `Codex` 作为 AuraEve 可调用的“外部编码能力”接进来，由 AuraEve 自主判断何时调用、调用哪个、是否复用会话、是否继续串联。

---

## 目标

### 核心目标

1. 让 AuraEve 能以统一方式调用 `Claude Code` 和 `Codex`。
2. 让主智能体把它们视为“能力”而不是零散命令。
3. 支持一次性执行与可复用会话两种模式。
4. 第一版本机优先运行，未来可以平滑扩展到远程节点。
5. 对外保留 ACP 风格的会话抽象，但第一版不实现完整 ACP bridge。

### 结果目标

接入完成后，AuraEve 应该可以：

- 在判断合适时调用外部编码智能体执行代码类或通用复杂任务。
- 自动选择 `claude` 或 `codex`，也允许显式指定。
- 为外部编码任务保留独立 session、cwd、状态和错误信息。
- 将结果以结构化工具结果的形式回注给主智能体继续决策。

### 非目标

以下内容不纳入本次设计范围：

1. 不实现完整 ACP 协议 server / stdio bridge。
2. 不实现 IDE 侧直接接入。
3. 不把外部编码智能体与现有 `subagent` 任务模型彻底统一成同一种底层执行器。
4. 不预置固定工作流，例如必须先 `Claude` 再 `Codex`。
5. 不在第一版做远程节点上的外部 coding runtime 分发。

---

## 总体方案

### 总体思路

采用“统一外部编码运行时 + 统一适配层 + AuraEve 内部工具入口”的分层方案：

1. 主智能体仍然是最终决策者。
2. 当主智能体认为需要外部编码能力时，通过一个新的工具入口请求外部编码执行。
3. 工具入口不直接调用具体 CLI，而是进入统一的 `ExternalAgentService`。
4. `ExternalAgentService` 再调用底层 `ExternalAgentRuntime`。
5. `ExternalAgentRuntime` 通过 adapter 适配底层 coding harness。

第一版推荐优先实现“统一 adapter”路线，尽量借鉴 OpenClaw 中 `acpx` 的能力模型，而不是分别手写两套 `claude` / `codex` 命令适配逻辑。这样做的核心收益是：AuraEve 依赖的是一套稳定接口，底层目标只是不同的 `agent target`。

### 为什么不直接做普通工具

如果只把 `Claude Code` 和 `Codex` 做成两个普通工具，短期看实现快，但很快会卡在这些问题上：

- 会话如何复用
- 工作目录如何保持
- 任务如何取消
- 状态如何查询
- 长时输出如何跟踪
- 失败后是销毁会话还是继续复用

这些问题本质上不是“工具调用问题”，而是“运行时管理问题”。因此这次设计必须把它们放进一个统一 runtime 层。

## 架构设计

### 分层结构

建议新增四层结构：

#### 1. 决策层

由 AuraEve 现有主智能体负责：

- 判断是否需要外部编码智能体
- 判断是偏向 `claude` 还是 `codex`
- 判断用一次性模式还是会话模式
- 判断拿到结果后是否继续追问、转交、总结或直接回复用户

#### 2. 服务层

新增 `ExternalAgentService`，负责：

- 目标 agent 解析
- 调用模式解析
- cwd 解析
- 本机 / 远程执行位置路由
- session 复用策略
- 调用 adapter 并收敛为统一结果

#### 3. 运行时层

新增 `ExternalAgentRuntime` 抽象，提供类似 ACP 的统一接口：

- `ensure_session`
- `run_turn`
- `cancel`
- `close`
- `get_status`
- `set_option`

这一层是 AuraEve 内部依赖的稳定契约。

#### 4. 适配层

第一版实现 `AcpxAdapter`，负责：

- 调起外部命令
- 创建 / 复用 coding harness session
- 把 AuraEve 请求翻译成底层命令参数
- 读取输出流并转成统一事件
- 屏蔽 `Claude Code` / `Codex` 的差异

### 模块建议

建议新增如下模块：

- `auraeve/external_agents/models.py`
- `auraeve/external_agents/runtime.py`
- `auraeve/external_agents/service.py`
- `auraeve/external_agents/registry.py`
- `auraeve/external_agents/store.py`
- `auraeve/external_agents/adapters/acpx_adapter.py`
- `auraeve/agent/tools/coding_agent.py`

现有模块只做有限改动：

- `auraeve/agent/tools/assembler.py`
- `auraeve/agent/context.py`
- `auraeve/agent_runtime/kernel.py`
- `auraeve/subagents/control_plane/orchestrator.py`
- `main.py`

## 数据模型设计

### ExternalAgentTarget

表示外部编码目标：

- `id`: `claude` | `codex`
- `kind`: `coding`
- `supports_session`: 是否支持持久会话
- `supports_cwd`: 是否支持工作目录
- `supports_cancel`: 是否支持取消
- `supports_options`: 支持哪些动态配置项

### ExternalSessionHandle

表示 AuraEve 视角下的外部会话：

- `session_id`
- `target`
- `mode`
- `cwd`
- `status`
- `created_at`
- `updated_at`
- `origin_session_key`
- `execution_target`
- `backend_session_ref`

其中：

- `session_id` 是 AuraEve 内部稳定 ID
- `backend_session_ref` 是底层 adapter/harness 的原生 session 标识

### ExternalRunRequest

表示一次运行请求：

- `task`
- `target`
- `cwd`
- `mode`
- `label`
- `timeout_s`
- `context_mode`
- `expected_output`
- `session_id`
- `execution_target`

### ExternalRunResult

表示一次运行结果：

- `status`
- `target`
- `session_id`
- `final_text`
- `summary`
- `artifacts`
- `raw_output_ref`
- `error`
- `usage`
- `suggested_next_action`

### ExternalRuntimeEvent

统一外部运行时事件：

- `session_started`
- `text_delta`
- `tool_event`
- `warning`
- `completed`
- `failed`
- `canceled`

即便第一版不完整暴露流式事件，也建议内部先按事件模型实现。

## 工具层设计

### 新工具：coding_agent

为主智能体新增一个统一入口工具 `coding_agent`。

它不是直接跑命令，而是调用 `ExternalAgentService`。

#### 参数建议

- `action`
  - `run`
  - `status`
  - `cancel`
  - `close`
- `task`
- `target`
  - `auto`
  - `claude`
  - `codex`
- `mode`
  - `oneshot`
  - `session`
- `cwd`
- `session_id`
- `label`
- `timeout_s`
- `expected_output`
  - `generic`
  - `summary`
  - `patch`
  - `review`
- `context_mode`
  - `summary`
  - `full_prompt`

#### 第一版建议支持的 action

1. `run`
2. `status`
3. `cancel`
4. `close`

先不做复杂的 `set_option` 对外暴露，但内部 runtime 接口保留。

### 主智能体使用原则

在系统提示或 tooling section 中为 AuraEve 补充明确原则：

1. 当任务涉及持续文件操作、复杂命令执行、较大代码改动时，可考虑 `coding_agent`。
2. `claude` 更偏分析、方案、审查。
3. `codex` 更偏实现、修改、修复。
4. 小任务优先直接使用当前本地工具，不必一律外包。
5. 若存在上下文连续的外部 session，优先复用。

这样可以实现“能力接入优先，使用方式由 AuraEve 自主判断”的目标。

## 全流程设计

### 流程 1：一次性执行

#### 步骤

1. 用户向 AuraEve 提出任务。
2. `RuntimeKernel` 正常组装上下文并驱动主智能体推理。
3. 主智能体判断当前任务适合交给外部编码智能体。
4. 主智能体调用 `coding_agent(action="run", mode="oneshot", target="auto", ...)`。
5. `coding_agent` 调用 `ExternalAgentService.run(...)`。
6. `ExternalAgentService` 解析目标，选出 `claude` 或 `codex`。
7. `ExternalAgentService` 通过 `ExternalAgentRuntime.ensure_session(...)` 创建临时 session。
8. 通过 `run_turn(...)` 执行任务。
9. `AcpxAdapter` 将请求转成底层命令并收集结果。
10. 运行结束后返回 `ExternalRunResult`。
11. 工具把结果以结构化文本或结构化 JSON 字符串返回给主智能体。
12. 主智能体基于结果决定是否直接回复、继续追问或再次调起其他能力。

### 流程 2：持久会话执行

#### 步骤

1. 主智能体调用 `coding_agent(..., mode="session")`。
2. `ExternalAgentService` 尝试查找可复用 session。
3. 如果 session 不存在，则创建新 session。
4. 如果 session 存在，则继续在同一个 session 上 `run_turn`。
5. 返回结果时带上 `session_id` 与状态摘要。
6. 后续若 AuraEve 判断上下文连续，可以继续对同一 session 发指令。

### 流程 3：取消与关闭

#### 取消

1. 主智能体或系统决定终止当前运行。
2. 调用 `coding_agent(action="cancel", session_id=...)`
3. `ExternalAgentService` 调用 runtime `cancel`
4. 更新 store 状态为 `aborted` 或 `canceling`

#### 关闭

1. 主智能体判断该 session 无需再保留。
2. 调用 `coding_agent(action="close", session_id=...)`
3. runtime 关闭底层会话
4. session store 标记为 `closed`

## Session Store 设计

### 为什么要有 store

即使第一版只在本机运行，也建议持久化 session 元数据。原因如下：

1. 支持复用外部会话。
2. 支持中断后恢复状态。
3. 支持 WebUI 观察。
4. 支持后续扩展为远程节点执行。

### 建议存储内容

建议新增独立 store，例如 `workspace/data/external_agent_sessions.json` 或独立 sqlite 表。

建议字段：

- `session_id`
- `target`
- `mode`
- `cwd`
- `origin_session_key`
- `backend`
- `backend_session_ref`
- `status`
- `created_at`
- `updated_at`
- `last_run_summary`
- `last_error`
- `execution_target`
- `node_id`

第一版建议优先用结构简单的本地 JSON store 或 sqlite 小表，避免和现有 `SessionManager` 对话历史混淆。

## 目标选择与路由策略

### target=auto 的选择逻辑

建议采用简单启发式，不做过度智能化：

#### 优先选择 claude 的情况

- 明确是方案设计
- 代码审查 / review
- 架构分析
- 风险评估
- 任务本身强调“分析、解释、评估、建议”

#### 优先选择 codex 的情况

- 明确是代码实现
- 文件修改
- bug 修复
- 测试补齐
- 命令执行与开发迭代

#### 复用优先规则

如果同一会话上下文已有仍健康的外部 coding session，且目标与任务连续，则优先复用已有 session，而不是强制重新选型。

### 执行位置路由

第一版固定走本机：

- `execution_target = local`

但 `ExternalAgentService` 内部接口保留未来扩展：

- `execution_target = local | remote`
- `node_id`

这样第二版支持远程节点时，不需要推翻工具语义。

## 适配层设计

### 为什么推荐统一 adapter

逐个对接 `Claude Code` 和 `Codex` 的原生命令行接口，短期可行，但长期问题明显：

1. 两套命令格式不同。
2. 会话命名规则不同。
3. 输出解析不同。
4. 错误语义不同。
5. 后续接第三个 agent 成本继续叠加。

因此第一版推荐引入统一 adapter 思路，优先实现 `AcpxAdapter`。AuraEve 的依赖对象是一套统一接口，底层目标由 adapter 处理。

### AcpxAdapter 的职责

1. 根据目标 agent 构建运行命令。
2. 确保或创建底层 session。
3. 把 AuraEve 请求翻译为底层 prompt 调用。
4. 读取 stdout / stderr。
5. 识别完成、失败、取消。
6. 输出 AuraEve 内部统一事件和结果。

### Adapter 的边界

Adapter 负责“如何对接外部进程”，但不负责：

- 业务目标选择
- 调度策略
- 用户会话路由
- 主智能体决策

这些应保留在 AuraEve 上层。

## 与现有子体系统的关系

### 关系定位

第一版不建议把外部编码智能体直接塞进现有 `TaskOrchestrator` 作为完全等价的 `subagent` 类型，而建议采用“并行体系、统一入口”的策略：

1. `subagent` 仍代表 AuraEve 自己控制的本地 / 远程子体。
2. `coding_agent` 代表对外部 coding runtime 的统一调用入口。
3. 两者都可被主智能体调用。

### 为什么不立刻统一

因为现有 `subagent` 系统当前重点是：

- 任务生命周期
- DAG
- 审批
- 本地 / 远程节点调度
- synthetic tool result 注入

而外部 coding runtime 的核心复杂度在于：

- 外部 session 维护
- 工作目录
- 命令适配
- 输出解析
- 权限 / 非交互失败

两者当前问题域并不完全一致。第一版先解耦实现，风险最小。

### 为统一预留的接口

虽然第一版不统一底层执行器，但可以预留：

- `execution_target`
- `target_kind`
- `session_id`
- `origin_session_key`

后续如果需要，再把 `coding_agent` 提升为 `TaskOrchestrator` 的一种执行后端。

## 错误处理设计

### 错误分类

建议把外部编码运行时错误映射为 AuraEve 可理解的几类：

#### 1. 启动错误

例如：

- adapter 命令不存在
- 依赖未安装
- 启动参数无效

处理策略：

- 快速失败
- 返回结构化诊断
- 不创建可复用 session，或回滚临时 session

#### 2. Session 初始化错误

例如：

- cwd 不存在
- session 无法创建
- 底层 backend 拒绝初始化

处理策略：

- 返回 `status=error`
- 清理临时会话
- 在 store 记录错误摘要

#### 3. 执行错误

例如：

- 外部智能体任务执行失败
- 返回错误事件
- 输出为空且退出异常

处理策略：

- 保留 session（如果可能）
- 允许后续继续或关闭
- 将错误映射为结构化结果返回主智能体

#### 4. 权限错误

这是第一版必须显式考虑的问题。外部 coding agent 通常涉及文件写入和命令执行，在非交互模式下容易因审批失败而终止。

处理策略：

- 将其标记为明确错误类别，如 `permission_denied_noninteractive`
- 不吞掉错误
- 提示主智能体这是“执行环境限制”，而不是普通编码失败

#### 5. 超时 / 取消 / 中断错误

处理策略：

- timeout 时保留 session 并标记为 `timeout`
- cancel 时优先执行底层取消命令
- close 时主动释放底层会话

### 错误对主智能体的暴露方式

主智能体应看到结构化结果，而不是单纯 stderr 文本。建议包含：

- `status`
- `error_type`
- `error_message`
- `retryable`
- `session_survived`

这样 AuraEve 才能自主判断：

- 是否换目标 agent 重试
- 是否改 cwd 继续
- 是否直接向用户解释环境问题

## 结果回注设计

### 原则

外部编码结果不能只是原始大文本。  
应当返回结构化结果给主智能体继续决策。

### 建议结果结构

建议 `coding_agent` 返回的核心内容包括：

- `target`
- `session_id`
- `status`
- `summary`
- `final_text`
- `artifacts`
- `error`
- `next_hint`

### 主智能体的后续决策

主智能体拿到结果后可以自行决定：

1. 直接回复用户
2. 要求同一 session 继续
3. 转交另一个外部 agent
4. 自己归纳总结
5. 放弃外部编码能力，改用本地工具继续

这正是本设计最重要的产品目标之一。

## 可观测性与 WebUI 设计

### 需要暴露的最小状态

WebUI 或运行时状态接口建议至少能看到：

1. 外部 session 列表
2. 每个 session 的 target / cwd / mode / status
3. 最近一次运行摘要
4. 最近一次错误
5. 是否仍可继续复用

### 日志建议

建议为外部编码能力单独打日志命名空间，例如：

- `external_agents/service`
- `external_agents/acpx_adapter`
- `external_agents/runtime`

方便区分：

- 主智能体推理问题
- 外部 coding runtime 问题
- 底层 adapter / 命令问题

## 测试策略

### 单元测试

必须覆盖：

1. target 自动选择规则
2. session store 的增删改查
3. `ExternalAgentService` 的路由逻辑
4. adapter 对成功 / 失败 / 超时 / 取消的解析
5. `coding_agent` 工具的参数校验

### 集成测试

建议覆盖：

1. 主智能体调用 `coding_agent(run)` 能获得结构化结果
2. `mode=session` 下同一 session 能复用
3. `cancel` / `close` 能更新状态
4. adapter 失败时主智能体能收到可理解错误

### 真实环境 smoke test

至少准备一条最小 live gate：

1. 创建一个 `codex` oneshot 任务
2. 让其返回固定文本
3. 校验结果进入 AuraEve 工具返回
4. 再创建一个 `claude` oneshot 任务做同样验证

第一版不要求大规模自动化 live test，但至少要有人工 smoke 路径。

## 分阶段实施建议

### Phase 1：本机能力接入

目标：

- `coding_agent` 工具可用
- `AcpxAdapter` 可驱动 `claude` / `codex`
- 支持 `run/status/cancel/close`
- 支持 session store

### Phase 2：会话复用与主智能体策略增强

目标：

- `target=auto` 更稳定
- session 复用策略更完善
- WebUI 可查看会话状态

### Phase 3：远程节点扩展

目标：

- 在远程 node 上运行外部 coding runtime
- `ExternalAgentService` 支持本机 / 远程路由

### Phase 4：ACP 兼容入口

目标：

- 在已有 runtime 之上补协议层
- 支持 IDE 或第三方 client 接入

---

## 风险与权衡

### 风险 1：底层 adapter 不稳定

说明：

如果统一 adapter 行为和预期不一致，会影响 AuraEve 整体体验。

缓解：

- 把 adapter 严格隔离在适配层
- 对输出和错误做统一封装
- 保持 service 层不依赖具体 stdout 文本格式

### 风险 2：非交互权限失败

说明：

这是第一版最现实的风险来源。

缓解：

- 明确建模错误类型
- 在配置中暴露权限策略
- 在 smoke test 中优先验证写文件和命令执行场景

### 风险 3：主智能体滥用外部编码能力

说明：

如果系统提示不清楚，主智能体可能对小任务也频繁调用外部 coding agent。

缓解：

- 在系统提示中明确使用原则
- 记录调用 telemetry，后续再优化策略

### 风险 4：过早和 subagent 系统深度耦合

说明：

如果第一版就把外部 coding runtime 并入 DAG / 审批 / 补偿主链，复杂度会明显上升。

缓解：

- 第一版只做独立 runtime 能力
- 仅保留与子体系统整合的接口预留

---

## 最终结论

本次推荐方案是：

1. 在 AuraEve 内部新增“外部编码运行时”能力层。
2. 通过统一 adapter 优先接入 `Claude Code` 与 `Codex`。
3. 以 `coding_agent` 工具形式暴露给主智能体。
4. 调用时机、目标选择、是否复用会话由 AuraEve 自主判断。
5. 第一版本机优先，预留远程执行与 ACP 兼容扩展点。

这一路径相比“直接加两个命令工具”更重一些，但它真正解决的是“能力接入”问题，而不是“工具包装”问题；相比“直接做完整 ACP bridge”，又能把第一版复杂度控制在 AuraEve 当前架构能稳定承载的范围内。
